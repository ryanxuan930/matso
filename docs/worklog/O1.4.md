## 決策與陷阱（之後維護的人需要知道的 why）
- **HotStateStore Protocol + 兩實作**：沿用專案 DI 風格（同 MonotonicClock）。RedisHotState 供正式；InMemoryHotState 供單元測試「不需 Redis」與 O1.6 確定性 replay（無 I/O）。
- **diff = 新增 + 變動欄位**（不處理欄位移除；單位熱狀態欄位集固定）。相同值的「更新」不算 diff。put_unit（新單位部署）整包欄位視為 diff（讓 client 學到新單位）。
- **Broadcaster protocol 簽章演進**：O1.3 的 `publish(tick)` → `publish(tick, diff)`（帶 STATE_DIFF payload）。這是我自己近期未合併的契約，內部演進可接受；同步更新 NoOpBroadcaster + Kernel + O1.3 test。
- **單位狀態寫入路徑**：O1.4 只讓 Kernel 持有 hot_state 並 drain+broadcast；「子系統實際寫單位狀態」屬 O3.4（movement 等），此卡不做（現階段 diff 為空亦正確）。Kernel 整合以「手動 update_unit 後 run_tick」測試驗證 drain+broadcast 串接。
- 單位移除（destroyed unit）未納入 diff——記 backlog，O3.x 處理。
