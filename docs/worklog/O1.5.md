## 決策與陷阱（之後維護的人需要知道的 why）
- **ADR 002**：inline zstd LONGBLOB + 8MB 護欄。Phase 1 快照 <1MB，遠低於 16MB packet 上限。
- **復原範圍界定**：O1.5 的 recover 保證「復原到最近 checkpoint 當下狀態」。若 checkpoint 之後還有 Ledger 事件（mid-interval crash），完整前滾需 O1.6 的確定性 replay（從 checkpoint 重跑）或 RNG 狀態序列化（O1.1 backlog）。recover 回傳 events_after_checkpoint，>0 時交由 O1.6 前滾。驗收測試在 checkpoint 當 tick 崩潰（events_after=0）→ recover 完全還原。
- **checkpoint 內容**：目前只含單位熱狀態（hot_state.get_all()）。RNG state / order queue 未納入——mid-interval 前滾才需要，記 backlog。
- **rollback 語意**：載入 target checkpoint → restore 熱狀態 → 寫 ROLLBACK 事件。post-rollback 的 Ledger 事件保留為歷史（append-only），live 狀態重置；分支/重模擬的完整語意屬 White Cell（O7.4）。
- **restore 不產生 diff**：復原/回滾時 hot_state.restore() 直接換掉 store 並清 pending（client 會走全量重同步，不靠增量）。
