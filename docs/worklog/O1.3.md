## 決策與陷阱（之後維護的人需要知道的 why）
- **TICK_OVERRUN 的牆鐘量測抽象化**：pseudocode 的 `perf_counter()` 是非決定性牆鐘。若 engine 直接呼叫，(1) 破壞 golden replay 可重現、(2) 違反「engine 不碰牆鐘」精神。故定義 `MonotonicClock` Protocol 注入；真實 `PerfCounterClock` 放 `core/app/runtime.py`（engine 外），replay/測試注入 `NullMonotonicClock`（永遠回 0 → 不會 overrun → 確定性）。
- **子系統統一回傳 `list[LedgerEvent]`**：pseudocode 只顯式 append 裁決事件，但 sensors/comms 也會產生事件；讓每個子系統方法回傳事件、Kernel 收集後**一次批次 append**（單 tick 單交易，效率佳且事件順序確定）。
- **事件順序（決定性）**：orders 裁決（依 drain 順序）→ movement → sensors → comms → logistics → triggers → (overrun)。固定順序確保 hash chain 可重現。
- **TICK_OVERRUN 診斷資料暫存於 `aiDecision` JSON 欄**（schema 無中性 detail 欄）——記入 backlog：事件類型變多前應檢討加一個中性 `detail` JSON 欄。此事件本質非決定性，僅在 overrun 時出現，不影響 stateHash。
