## 決策與陷阱（之後維護的人需要知道的 why）
- **從 tick 0 重跑**：DeterministicRNG 由 (master_seed, stream_id) 建構，重跑即重現整個序列，不需 RNG state 序列化（O1.1 backlog 無需在此解）。
- **示範子系統用整數格點亂步**：PCG64 的整數串跨平台/版本穩定；避免浮點 uniform 的格式化在 mac 記錄、Linux CI 執行時產生 hash 差異。final state 全整數 → canonical_json 穩定。
- **rerecord 走環境變數 + subprocess**：golden 測試在 `MATSO_RERECORD_GOLDEN` 設定時「寫入」而非「斷言」。ops/tools/rerecord_golden.py 只是 subprocess 跑 pytest（不 import 測試碼），保持 mypy 乾淨、ops 與測試解耦。
- **真實 order 序列**：O3.1（order pipeline）尚未產生 ORDER_ISSUED 事件，故現階段想定以 (seed + 確定性子系統) 定義；O3.1 後可加入「讀 Ledger order 序列」的想定，harness 已預留 build_kernel 工廠可注入 scripted OrderSource。
- **replay 用 InMemoryHotState + NullMonotonicClock + null sink**：純記憶體、無 I/O、無牆鐘 → 完全確定性；Ledger 非 stateHash 目標，故丟棄事件。
